Overview

The problem, the approach, and the verdict.

The Problem

SAC (Soft Actor-Critic) fails to learn in DeepUnity. PPO works fine. Critic loss looks healthy but the actor reward plateaus near random baseline. Alpha sweeps and gradient rewrites didn't help.

The Approach

Two AI agents (Claude Code + Codex) collaborated via a shared file. Claude focused on optimizer/config analysis. Codex built numerical gradient probes with finite differences.

The Verdict

The gradient math is perfect (cos=1.0). The bugs are: accidental weight decay, inherited PPO gradient clipping, wrong LR scheduler, and critically low update-to-data ratio (50x below standard). Dense updates improved smoke bandit reward by 49%.

Bugs Found

Five bugs identified, ranked by severity. The top two are the root cause.

CRITICAL FIXED
SACTrainer.cs:74-75

Actor AdamW weight_decay = 0.01 (unintentional default)

The optim_mu and optim_sigma optimizers are created with new AdamW(...) without specifying weight_decay, inheriting AdamW's default of 0.01. The critic correctly sets it to 0.

BEFORE (Buggy)
optim_mu = new AdamW(params, lr: hp.actorLR); // weight_decay defaults to 0.01
AFTER (Fixed)
optim_mu = new AdamW(params, lr: hp.actorLR, weight_decay: 0f);
Impact: Every gradient step, actor parameters are multiplied by (1 - lr * 0.01). Over 100K steps: 26% parameter shrinkage. Over 1M steps: 95% shrinkage. TD3 and DDPG are unaffected (they use Adam with wd=0 or explicitly set wd=0).
CRITICAL FIXED
SACTrainer.cs:392,397

Actor gradient clipping inherited from PPO (maxNorm = 0.5)

The hyperparameters file states "defaults are for PPO". The SAC inspector hides maxNorm from the UI but never zeros it, so SAC inherits PPO's clip value of 0.5. Even PPO itself has commented out mu-network clipping.

BEFORE (Buggy)
model.muNetwork.Backward(-dJ_dMu); if (hp.maxNorm > 0f) optim_mu.ClipGradNorm(hp.maxNorm); optim_mu.Step();
AFTER (Fixed)
model.muNetwork.Backward(-dJ_dMu); optim_mu.Step();
Impact: Actor gradient norm is capped at 0.5 every step. SAC is the only off-policy algorithm that inherited this — TD3 and DDPG never reference ClipGradNorm.
HIGH TUNABLE
HyperParameters.cs:85

50:1 update-to-experience ratio

The canonical SAC does 1 gradient update per 1 env step (1:1). DeepUnity defaults to updateInterval=50, updatesNum=1, giving a 1:50 ratio. Each experience gets sampled ~0.43 times before eviction vs ~64 times in the standalone.

HIGH DESIGN
SACTrainer.cs:135-136

Replay buffer batch deletion (25% at a time)

When the replay buffer fills, RemoveRange(0, Count/4) deletes 250K experiences at once. The standalone uses a ring buffer with 1-at-a-time circular overwrite. Batch deletion creates distribution shifts and wastes unsampled data.

MEDIUM FIXED
AgentBehaviour.cs:708,713,723

Save/load copy-paste bugs (build path only)

Three serialization errors in the #if !UNITY_EDITOR save path: Q1 and Q2 both saved from vNetwork, Sigma saved from muNetwork. Editor save path is correct.

MEDIUM FIXED
SoftPlus.cs

SoftPlus deserialization null reference

Found by Codex while building the stochastic probe. The SoftPlus activation could reach Predict/Backward with an invalid internal tensor after asset deserialization. Hardened with runtime state repair.

Key Takeaway

Raising the update-to-data (UTD) ratio from 0.02 to 1.0 improved reward by 49% on a trivial one-step bandit. The default DeepUnity SAC cadence is 50x below the canonical SAC standard (SpinningUp, CleanRL, stable-baselines3).

User Test Diagnostic

BalanceBall SAC with weight_decay + grad clipping fixes, but still using default 50:1 cadence. The critic massively overestimates.

Symptom: J rises to 2.47 (98% of theoretical max 2.5) but actual episode length stays near random. The critic thinks the policy is nearly optimal when it's actually terrible. Entropy term stays flat at ~0.27 (never decreases). This is classic Q-value divergence from insufficient update frequency.

Additional Bugs Found

Discovered during the broader off-policy trainer audit.

CRITICAL FIXED
SACTrainer.cs, HyperParameters.cs

Low UTD ratio: 50x below canonical SAC standard

Default updateInterval=50 with updatesNum=1 gives 1 gradient step per 50 env transitions (UTD=0.02). Standard SAC uses UTD=1.0. Proven by ablation: 0.59 vs 0.88 reward.

HIGH FIXED
SACTrainer.cs:91, TD3Trainer.cs:41

LR Scheduler total_iters uses env steps instead of gradient steps

Both SAC and TD3 set total_iters = maxSteps but the scheduler steps per gradient update. Should use maxSteps * updatesNum / updateInterval like DDPG does. Fixed by Codex.

MEDIUM FIXED
SoftPlus.cs

SoftPlus deserialization null reference

The SoftPlus activation could reach Predict/Backward with an invalid internal psi tensor after asset deserialization. Fixed by Codex with runtime self-repair via EnsureState().

Recommendations

Additional improvements beyond the critical fixes.

Increase Update Frequency

Set updatesNum = 4-8 or reduce updateInterval to improve the update-to-experience ratio.

The canonical SAC uses a 1:1 ratio. Current default is 1:50.

Replace Replay Buffer

Switch from List with batch RemoveRange to a ring buffer with circular overwrite.

Prevents 250K-at-a-time data loss and distribution shifts.

SAC-Specific Defaults

Consider SAC-specific initialization in Initialize() to override PPO-oriented defaults (maxNorm, etc.).

Prevents future accidental inheritance from shared HyperParameters.

Softplus Overflow Guard

Add x > 20 ? x : log(1+exp(x)) guard to Tensor.Softplus for numerical stability.

The standalone has this guard; Unity doesn't. Low risk but good practice.
